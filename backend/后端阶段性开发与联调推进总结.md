# StudyQuest 后端阶段性开发与联调推进总结

## 1. 整体代码分层与架构设计

目前Java后端采用了Controller-Model-Repository三层结构。整体数据的流转链条是：

Android前端请求 -> Java 后端网关拦截 -> RestTemplate转发呼叫Python AI端 -> 拿到响应后按业务类型进行MySQL异步落库 -> 结果原样返回给前端。

### 目前把Service层留空
在目前的架构设计下，没有去强行堆砌Service层的代码。因为项目属于典型的 **AI Agent 驱动架构**。像最核心的意图分类、多模态OCR识别以及向量库（Qdrant）的RAG检索这些高负载、重型的计算逻辑，全部内聚在Python 那边。
Java后端现阶段的本质定位是**“高性能 API 网关 + 关系型数据持久化引擎”**。Controller在拿到Flask的结构化数据后，业务已经计算完毕，直接调用Repository就能一键落库，没有复杂的本地Java业务需要二次组装。保留Service目录，是为了后续扩展纯本地业务（如JWT权限校验、多表聚合统计或本地事务控制）留出的干净切面。

## 2. 已实现的业务逻辑与数据落库方案

目前项目通过一套统一的 `/api/studyquest` 入口，通过请求体里的 `request_type` 做动态路由，已经彻底调通了以下核心电路：

### 资料上传与向量化（upload分支）
* **逻辑**：用户在手机端上传数学错题或笔记图片，Java转发请求后，Python端通过 Doubao-Seed-2.0-pro视觉大模型做多模态OCR提取，利用text-embedding-3-small转化为1536维向量，最终存入远程云端Qdrant向量库。
* **现状**：此链路已部分跑通，目前在测试中远程Qdrant的 `studyquest_knowledge` 集合里已经成功注入了61个知识点分块，但RAG检索阈值需调整，不然返回null字段，后续会再进行测试。

### AI 智能答疑（qa分支）
* **逻辑**：基于 DeepSeek-Chat 核心，根据用户提问在向量库里做垂直领域的 RAG 检索。系统会根据关键词自动触发 hint（反问提示）、explain（分步详解）或 review（易错点总结）三种应答模式。
* **落库**：Java 接收到 Python 回应后，会自动剥离出大模型的回答内容，并将提问、解答、应答模式和用户 ID 组装，同步写入本地 MySQL 的 `qa_logs` 表，为后面的薄弱点复盘打下了数据根基。

### 学习关卡规划（plan分支）
* **逻辑**：用户输入目标，AI拆解为20-50分钟的原子任务，Java端写好了解析逻辑。接收到数据后会自动提取 `session_id` 存入会话表，并用一个for循环遍历整个复杂嵌套的任务数组，批量将所有关卡级联写入 `tasks` 表。

### 关卡清单查询与状态打卡（tasks独立控制层）
* 针对Android同学的联调需求，在 `TaskController` 里加了 `@CrossOrigin(origins = "*")` 支持全局跨域，并落地了两个纯本地 API：
  1. `GET /api/tasks/session/{sessionId}`：根据会话 ID 秒级调取所有关卡状态，供手机端渲染任务清单树。
  2. `PUT /api/tasks/{id}/status?status=completed`：支持关卡实时状态更新（打卡完成），状态的变更后续会直接作为 Review 模块进行薄弱点分析的数据源。

### 闲聊与报告的透传设计（chat/review分支）
* **chat 分支**：由于日常闲聊是非结构化的高频交互，为了防止MySQL的I/O吞吐量被垃圾数据撑爆，Java保持无状态转发。
* **review 分支**：复盘报告包含复杂的知识点加权矩阵和雷达图数据，属于强计算、重渲染的场景。Python算好后，Java选择直接透传其JSON结构给前端。这种设计避免了在本地关系型数据库里做恶的多表联查（JOIN），实现了前后端解耦。

## 3. 关键技术攻坚

### 跨服务熔断防御机制

* **针对上次测试出现的问题的解决**：在 `StudyQuestController` 接收端加了一道防御性拦截。一旦捕捉到 `data == null`，Java立即启动本地安全熔断，强行将响应体修正为success = false，并返回一句提示，提前拦截后续操作。

### 历史遗留测试代码的平稳升级
* 针对早期直连Python旧路由的 `GET /api/tasks/generate` 独立测试接口进行了定向重构，将原本对不上的 `stages` 和 `desc` 字段升级对齐为现代架构的 `tasks` 与 `description`，并融入了虚拟Session安全绑定，目前作为后端单点调试的隔离备用端点存在。
